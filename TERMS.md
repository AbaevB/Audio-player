# Техническое задание «Приложение „Аудиоплеер“»

## Что нужно сделать

Вам нужно реализовать клиентскую часть приложения «Аудиоплеер» в соответствии [с макетом.](https://www.figma.com/design/lO9aaBqUdW6vp1N33iydRY/TypeScript)

Для работы вам потребуется [бэкенд](https://drive.google.com/file/d/18H9bv2FUW6UvR7iKEU0f5p1qaBEmg2Q5/view). Разверните локально проект express-backend, установите все зависимости командой npm install (либо используя пакетный менеджер, подходящий для вашей ОС). Далее вы можете запустить сервер. Перейдите в проект, из него запустите сервер, используя команду start.

### Описание работы API

Base URL:(<http://localhost:8000>)
API Base Path:(<http://localhost:8000/api>)

### Endpoints

#### Регистрация нового пользователя

```typescript
 URL: /register
 Method: POST
```

Описание: регистрирует нового пользователя. Проверяет, существует ли пользователь с таким именем, и создаёт запись с хешированным паролем.

Пример запроса:

```ts
POST /api/register HTTP/1.1
 Content-Type: application/json

 {
 "username": "user1",
 "password": "securepassword"
 }
```

Пример ответа (успешная регистрация):

```js
{
 "message": "пользователь успешно добавлен",
 "user": {
 "username": "user1"
 }
 }
```

Пример ответа (ошибка):

```js
{
 "message": "пользователь уже существует"
 }
```

#### Авторизация пользователя.

```js
URL: /login
 Method: POST
```

Описание: авторизует пользователя. Проверяет имя пользователя и пароль, и в случае успеха возвращает токен для работы с API.

Пример запроса:

```js
POST /api/login HTTP/1.1
 Content-Type: application/json
 {
 "username": "user1",
 "password": "securepassword"
 }

```

Пример ответа (успешная авторизация):

```js

{
 "message": "авторизация прошла успешно",
 "token": "eyJhbGciOiJIUzI1NiIsInR5..."
 }

```

Пример ответа (ошибка):

```js
{
 "message": "произошла ошибка при авторизации — неверные данные"
 }
```

#### Получение списка композиций.

```js
URL: /tracks
 Method: GET
```

Описание: возвращает список всех доступных музыкальных композиций.

Пример запроса:

```
GET /api/tracks HTTP/1.1
 Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5...
```

Пример ответа:

```js
[
 {
 "id": "1",
 "title": "Song A",
 "artist": "Artist A"
 },
 {
 "id": "2",
 "title": "Song B",
 "artist": "Artist B"
 }
 ]
```

#### Работа с избранными композициями.

- Получение списка избранного.

```js
URL: /favorites
 Method: GET
``` 

Описание: возвращает список треков, добавленных в избранное пользователем.

Пример запроса:

```js
GET /api/favorites HTTP/1.1
 Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5...
```

Пример ответа:

```js
[
 {
 "id": "1",
 "title": "Song A",
 "artist": "Artist A"
 }
 ]
``` 

- Добавление композиции в избранное.

```js
URL: /favorites
 Method: POST
```

Описание: добавляет трек в список избранного пользователя.

Пример запроса:

```js
POST /api/favorites HTTP/1.1
 Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5...
 Content-Type: application/json
 {
 "trackId": "1"
 }
```

Пример ответа:

```js
{
 "message": "композиция добавлена в избранное"
 }
```

- Удаление композиции из избранного.

```js
 URL: /favorites
 Method: DELETE
```

Описание: удаляет трек из списка избранного пользователя.

Пример запроса:

```js
DELETE /api/favorites HTTP/1.1
 Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5...
 Content-Type: application/json
 {
 "trackId": "1"
 }
```

Пример ответа:

```js
{
 "message": "композиция убрана из избранного"
 }
```

### Авторизация

Для всех защищённых эндпоинтов (/favorites, /tracks) требуется токен. После успешного логина токен возвращается в ответе. Токен должен быть передан в заголовке Authorization в формате:

```js
Authorization: Bearer <token>
```

Фундаментально вашу задачу можно разделить на следующие структурные блоки:

- Визуальная составляющая. Сюда входит вёрстка компонентов приложения, их оформление, визуальные переходы между страницами, анимации.

- Клиентская логика. Вам нужно обеспечить правильное взаимодействие всех компонентов приложения, оптимизировать эти процессы, используя базовые принципы разработки и архитектуры.

- Взаимодействие с сервером. Здесь ваша зона ответственности находится только в пределах клиентской части. Нужно грамотно настроить логику взаимодействия с API, работу с полученными данными и их отправкой на сервер.

## Вёрстка компонентов

- Адаптивность приложения под разные виды устройств.

- Корректная работа в браузерах Chrome, Firefox, Safari, Opera, Edge (достаточно самых свежих версий).

- Валидный CSS и HTML. Для проверки можете использовать любой удобный валидатор.

- Допускается использование препроцессоров.

- Именование классов в соответствии с методологией БЭМ. Либо можете использовать какую-нибудь библиотеку компонентов — в таком случае напишите пару слов, почему выбрали именно её для вашего проекта.

## Разработка логики приложения

- Для построения модулей приложения можете пользоваться двумя подходами: создать статичные фрагменты HTML в виде строки и добавлять в них вкрапления логики на TypeScript либо генерировать все элементы на лету по мере выполнения скрипта.

- Избегайте прямого использования низкоуровневых методов работы с DOM. Выберите подходящую для этих целей библиотеку (рекомендуем Redom).

- Обязательная и осмысленная типизация на проекте. Использование any должно быть исключено.

- Логически настроенная навигация в приложении. Кнопки перехода между экранами.

## Необходимые функции приложения (медиа)

1. Прослушивание аудиофайла.

2. Перемотка аудиофайла в обе стороны (плюс-минус 10 секунд) с помощью клавиатуры или мыши.

3. Перемотка аудиофайла непосредственно на временной шкале по клику на её участок.

4. Переход к следующему файлу.

5. Переход к предыдущему файлу.

6. Функция паузы.

7. Добавление трека в избранное.

## Необходимые функции приложения (базовое)

1. Авторизация в приложении.

2. Регистрация в приложении.

## Необходимые функции приложения (навигация и иное)

1. Просмотр списка треков с пагинацией. Для десктопной версии можно реализовать классическую пагинацию, для мобильной — в формате ленивой подгрузки, когда новые треки отображаются по мере приближения к концу экрана.

2. Карточка трека с основной информацией о нём.

3. Страница профиля пользователя с базовой информацией и аватаром-заглушкой

## Советы и рекомендации

1. Используйте знания, полученные на курсе, и отталкивайтесь от того базового набора технологий, который уже изучили. Однако поиск новых решений и библиотек — неотъемлемая часть работы разработчика, поэтому будет здорово, если найдёте свои инструменты под конкретные потребности.

2. Как уже говорилось выше, аудиоплеер является очень популярным типом приложений, поэтому и набор инструментов для вас столь же богат. Полезно будет изучить и уже готовые примеры с открытым исходным кодом.

3. Добавьте творческий подход. Важно не урезать изначальное ТЗ и следовать макетам, но ничего не мешает вам дополнить приложение фишками в плане UI, микрологикой для большей отзывчивости плеера на действия пользователя и другими находками и решениями.

Хорошей практикой является использование так называемых линтеров — специальных плагинов, которые в фоновом режиме следят за соответствием кода общепринятым стандартам. [Пример линтера](https://www.npmjs.com/package/eslint-plugin-filenames), который вы можете использовать.
